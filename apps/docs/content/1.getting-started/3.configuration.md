# Configuration

Agent Please uses a single `WORKFLOW.md` file as its configuration source. This file lives in the root of your target repository and contains YAML front matter for settings and a Liquid template for the agent prompt.

## File Structure

```yaml [WORKFLOW.md front matter]
platforms:
  github:
    kind: github
    api_key: $GITHUB_TOKEN
    owner: myorg

projects:
  - platform: github
    project_number: 1

polling:
  interval_ms: 30000

workspace:
  root: ./workspaces

agent:
  max_concurrent_agents: 10
  max_turns: 20
```

The prompt template follows the YAML front matter:

```markdown [WORKFLOW.md body]
You are working on issue {{ issue.title }}.

{{ issue.body }}
```

## Platform Configuration

Platforms define how Agent Please connects to your issue trackers. Each platform has a unique key name.

### GitHub Projects v2

```yaml
platforms:
  github:
    kind: github
    owner: myorg
    # PAT authentication
    api_key: $GITHUB_TOKEN
    # OR GitHub App authentication
    app_id: $GITHUB_APP_ID
    private_key: $GITHUB_APP_PRIVATE_KEY
    installation_id: $GITHUB_APP_INSTALLATION_ID
```

### Asana

```yaml
platforms:
  asana:
    kind: asana
    api_key: $ASANA_ACCESS_TOKEN
```

## Project Configuration

Projects reference a platform by name and specify which project to poll:

### GitHub

```yaml
projects:
  - platform: github
    project_number: 1
```

### Asana

```yaml
projects:
  - platform: asana
    project_gid: '1234567890'
```

### Status Configuration

Each project can customize which statuses are considered active, terminal, or watched:

```yaml
projects:
  - platform: github
    project_number: 1
    active_statuses: [Todo, In Progress, Merging, Rework]
    terminal_statuses: [Closed, Cancelled, Done]
    watched_statuses: [Human Review]
```

**Defaults (GitHub):** active: `Todo, In Progress, Merging, Rework` | terminal: `Closed, Cancelled, Done` | watched: `Human Review`

**Defaults (Asana):** active: `To Do, In Progress` | terminal: `Done, Cancelled`

### Filters

Filter which issues the orchestrator picks up:

```yaml
projects:
  - platform: github
    project_number: 1
    filter:
      assignee:
        - agent-please-bot
      label:
        - agent-ready
```

## Environment Variable Resolution

Config values matching `$ENV_VAR` are resolved from `process.env` at startup. This keeps secrets out of the file:

```yaml
platforms:
  github:
    api_key: $GITHUB_TOKEN # resolved from process.env.GITHUB_TOKEN
```

## Polling Configuration

```yaml
polling:
  interval_ms: 30000 # default: 30000 (30 seconds)
```

## Workspace Configuration

```yaml
workspace:
  root: ./workspaces # base directory for issue workspaces
```

Each issue gets a dedicated subdirectory under `workspace.root`. The orchestrator validates all workspace paths to prevent path traversal.

## Agent Configuration

```yaml
agent:
  max_concurrent_agents: 10 # default: 10
  max_turns: 20 # default: 20
  max_retry_backoff_ms: 300000 # default: 300000 (5 minutes)
```

### Per-State Concurrency

```yaml
agent:
  max_concurrent_agents: 10
  max_concurrent_agents_by_state:
    In Progress: 3
    Human Review: 2
```

## Server Configuration

```yaml
server:
  port: 3000
  webhook:
    secret: $GITHUB_WEBHOOK_SECRET
```

## Workspace Hooks

```yaml
hooks:
  after_create: 'git clone $REPO_URL . && bun install'
  before_run: 'git pull origin main && bun install'
  after_run: bun run test
  before_remove: "echo 'Cleaning up'"
```

## Prompt Template

The body of `WORKFLOW.md` (below the front matter) is a Liquid template rendered with issue context:

| Variable | Description |
|----------|-------------|
| `{{ issue.title }}` | Issue title |
| `{{ issue.body }}` | Issue body/description |
| `{{ issue.number }}` | Issue number |
| `{{ issue.url }}` | Issue URL |
| `{{ issue.state }}` | Current project state/column |

### Conditional Blocks

```markdown
{% if issue.labels contains "bug" %}
This is a bug fix. Focus on reproducing and fixing the issue.
{% else %}
Implement the feature as described.
{% endif %}
```

## Live Reload

Agent Please watches `WORKFLOW.md` for changes. When the file is modified, the config is re-parsed automatically. Invalid configs are rejected and the last known-good config is retained.
